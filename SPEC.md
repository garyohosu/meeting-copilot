# 1. 目的

Pixel Recorder 等の文字起こしサービスで取得した文字起こし結果をもとに、会議後の業務を効率化するアプリを作成する。

本アプリの主目的は以下とする。

1. 文字起こしデータを取得する
2. AIで要約する
3. 議事録を生成する
4. メール文を生成する
5. 必要に応じて TODO や決定事項を抽出する

将来的には、会議中のリアルタイム支援も目指す。

---

# 2. 開発方針

本アプリは以下の 2 段階で開発する。

## Phase 1
会議後処理アプリ

- Pixel Recorder の文字起こし結果 URL または文字起こし本文を入力
- 内容を取得
- AIで整形
- 議事録、メール文、要約、TODO を生成
- コピーまたは保存できる

## Phase 2
会議中リアルタイム支援

- 音声または逐次文字起こしを受信
- 話題要約をリアルタイム表示
- 話し方アドバイスを提示
- 会議中に出た疑問を検索
- 補助メモや提案を表示

本仕様書では、Phase 1 を必須実装対象、Phase 2 を拡張対象とする。

# 4. 用語定義

## 4.1 Pixel Recorder 文字起こし
Pixel Recorder 等の音声文字起こしサービスで取得した文字起こしデータ。
Pixel Recorder 以外のサービス（例: Otter.ai, Notta, Microsoft Teams 等）の出力も入力対象とする。

## 4.2 文字起こしURL
Pixel Recorder 共有機能などにより発行された、文字起こし内容にアクセスできる URL。

## 4.3 議事録
会議内容を整理した正式文書。主に以下を含む。
- 会議名
- 日時
- 出席者
- 議題
- 内容要約
- 決定事項
- 宿題 / TODO
- 次回予定

## 4.4 メール文
会議内容をもとにした報告メール、依頼メール、共有メール等の文面。

# 7. 入力仕様

## 7.1 入力方式
以下の2方式をサポートする。

### A. URL入力
ユーザーが文字起こし共有URLを入力する。

### B. 本文貼り付け
ユーザーが文字起こし本文を直接貼り付ける。

---

## 7.2 入力項目

### 必須項目
- 入力種別（URL / 本文）
- 文字起こしURL または 文字起こし本文

### 任意項目
- 会議名
- 会議日時
- 出席者
- 出力言語
- メール用途
- 議事録テンプレート種別
- 要約の長さ
- 追加指示（初回生成用・全体共通）

---

## 7.3 入力バリデーション

### URL入力時
- URL形式であること
- 空文字でないこと
- アクセス可能であること（取得処理時）

### 本文入力時
- 空文字でないこと
- 最低文字数を満たすこと（例: 20文字以上）

### 共通
- 文字数上限: 50,000文字以内（超過時はエラー表示、自動分割しない）
  - エラー文言例: 「入力文字数が上限を超えています。50,000文字以下にしてください」
- 推奨文字数: 20,000文字以内
- 許可されていない形式は拒否すること

# 9. 機能要件

# 9.1 Phase 1 必須機能

## F-001 URL入力機能
ユーザーが文字起こしURLを入力できること。

## F-002 本文貼り付け機能
ユーザーが文字起こし本文を直接貼り付けできること。

## F-003 文字起こし取得機能（補助機能）
URL入力時、URL先から文字起こし本文を取得できること。
取得対象は認証不要で公開されているURLのみとする。
認証が必要なURL、または取得に失敗した場合は、取得失敗として扱い本文貼り付けへ誘導すること。

## F-004 AI要約機能
取得した文字起こし本文をAIで要約できること。

## F-005 議事録生成機能
文字起こし内容から議事録を生成できること。

## F-006 メール文生成機能
文字起こし内容からメール文を生成できること。

## F-007 TODO抽出機能
文字起こし内容からTODOを抽出できること。

## F-008 決定事項抽出機能
文字起こし内容から決定事項を抽出できること。

## F-009 結果コピー機能
各生成結果をコピーできること。

## F-010 再生成機能
各出力タブ（要約・議事録・メール文・TODO）ごとに、追加指示を入れて個別に再生成できること。
再生成時は、初回生成時の全体追加指示に加えて、対象タブ専用の追加指示を適用できること。
全体一括再生成は任意機能とする。

## F-011 エラー表示機能
取得失敗、解析失敗、AI処理失敗時にエラー内容を表示できること。

## F-013 保存機能
生成結果を Markdown 形式でローカルファイルとして保存できること。
Phase 1 では、要約・議事録・メール文・TODO・決定事項を1つの Markdown にまとめて保存すること。
現在表示中タブのみの保存は将来拡張とする。

# 10. 非機能要件

## 10.1 操作性
- 初心者でも使いやすいUIであること
- 主要操作（入力・実行・結果確認）は3ステップ以内を目標とする
- 1画面で入力から出力まで完結するUIが望ましい
- 結果をすぐコピーできること

## 10.2 性能
- 本文入力時は数秒から十数秒程度で結果表示を目指す
- 長文でも安定して処理できること
- 必要に応じて分割処理できること

## 10.3 可用性
- URL取得に失敗しても本文貼り付けで代替できること
- AI処理失敗時にリトライ可能であること

## 10.4 セキュリティ
- 入力された文字起こし内容を不用意に外部保存しないこと
- APIキーは環境変数で管理し、ソースコードに埋め込まないこと
- 本アプリは OpenAI API を利用するため、入力された文字起こし本文・追加指示・再生成指示は第三者サービスである OpenAI API に送信される
- 文字起こし本文・生成結果はセッション中のみ保持し、サーバー側に永続保存しないこと
- アプリ自身は、送信データをサーバー側に永続保存しないこと
- ログにはエラー概要のみ記録し、本文は含めないこと
- 機密情報・個人情報を含む会議データを扱う場合は、利用前に組織内ルールおよび利用者判断に基づき送信可否を確認すること
- 個人情報や機密情報の取り扱いに配慮すること

## 10.5 保守性
- URL取得処理
- AI整形処理
- 出力整形処理
を分離した構造にすること

# 12. 業務フロー

## 12.1 基本フロー
1. ユーザーがURLまたは本文を入力
2. システムが文字起こし本文を取得
3. システムが本文を整形
4. AIが要約・議事録・メール文・TODOを生成
5. ユーザーが内容を確認
6. ユーザーがコピーまたは保存

---

## 12.2 URL取得失敗フロー
1. URLアクセス失敗
2. エラーメッセージ表示
3. 本文貼り付けを案内
4. ユーザーが本文を貼り付けて再実行

---

## 12.3 AI生成失敗フロー
1. AI呼び出し失敗
2. エラーメッセージ表示
3. 再試行または入力修正を案内

# 14. AIプロンプト方針

AIには以下の役割を与える。

- 会議内容の整理者
- 議事録作成者
- ビジネスメール作成者
- TODO抽出者

AIへの主な指示方針:

- 冗長な会話を整理する
- 発言内容を要点化する
- 推測しすぎない
- 不明点は不明と明示する
- 担当者や期限は本文に根拠がある場合のみ出力する

---

# 15. 制約事項

- Pixel Recorder の共有URLは認証が必要な場合があり、認証不要のURLのみを取得対象とする
- 認証が必要なページは取得難易度が高い
- URL取得が不安定な場合は本文貼り付け方式を主運用とする
- 会話中に明示されていない担当者や期限は推定精度に限界がある
- OpenAI API を利用するため、外部送信が許可されない会議データには適用できない場合がある

# 20. 実装メモ

最初のPoCでは、URL自動取得にこだわりすぎず、本文貼り付けでも価値が出る構成にすること。
まず「会議後の議事録・メール作成を大幅に楽にする」ことを成功条件とする。
リアルタイム支援は別フェーズで切り出すこと。
